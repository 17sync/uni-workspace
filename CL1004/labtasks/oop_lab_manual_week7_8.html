üåü Part 1: Static Members - The Shared Resources

What Are Static Members?

Imagine a university where every student has their own ID, name, and GPA (these are instance variables). But what if we want to know how many total students are enrolled? That's where static members come in!

Real-World Analogy: The Scoreboard üèÄ

Think of a basketball game:

  • Each player has their own jersey number, position, and personal score (non-static)
  • The team's total score is displayed on ONE scoreboard that ALL players contribute to (static)
  • You don't need to look at a specific player to see the team score - it belongs to the team as a whole!

Static Data Members

Static data members are shared by ALL objects of a class. There's only ONE copy of a static variable, no matter how many objects you create!

üìÑ BankAccount.h - Without Static Members
// Without static - each account has its own counter (WRONG!)
class BankAccount {
private:
    int accountNumber;
    double balance;
    int totalAccounts;  // ‚ùå This creates a separate counter for each account!

public:
    BankAccount(double bal) {
        balance = bal;
        totalAccounts++;  // This only increases THIS object's counter
    }
};
üìÑ BankAccount.h - With Static Members (CORRECT!)
class BankAccount {
private:
    int accountNumber;
    double balance;
    static int totalAccounts;  // ‚úÖ Shared by ALL accounts!

public:
    BankAccount(double bal) {
        balance = bal;
        accountNumber = ++totalAccounts;  // Increments the shared counter
    }

    void displayAccount() {
        cout << "Account #" << accountNumber << ": $" << balance << endl;
    }

    static int getTotalAccounts() {  // Static method
        return totalAccounts;
    }
};
Notice the static keyword before the declaration. This tells C++ that this variable is shared across ALL instances of the class!
üìÑ BankAccount.cpp - Initializing Static Members
#include "BankAccount.h"

// CRITICAL: Static members MUST be initialized outside the class!
int BankAccount::totalAccounts = 0;  // Initialize to 0

// The rest of your implementation goes here
Static data members MUST be defined (initialized) outside the class definition! If you forget this, you'll get a linker error. Think of it as "activating" the shared resource.
üìÑ main.cpp - Using Static Members
#include <iostream>
#include "BankAccount.h"
using namespace std;

int main() {
    cout << "Initial accounts: " << BankAccount::getTotalAccounts() << endl;
    // Output: 0 (called without any object!)

    BankAccount acc1(1000.0);
    cout << "After 1st account: " << BankAccount::getTotalAccounts() << endl;
    // Output: 1

    BankAccount acc2(2000.0);
    BankAccount acc3(1500.0);
    cout << "After 3 accounts: " << BankAccount::getTotalAccounts() << endl;
    // Output: 3

    acc1.displayAccount();  // Account #1: $1000
    acc2.displayAccount();  // Account #2: $2000
    acc3.displayAccount();  // Account #3: $1500

    return 0;
}

Static Methods

Static methods are functions that belong to the class itself, not to any specific object. They can ONLY access static members!

Real-World Analogy: The Information Desk üìû

Think of a library:

  • Non-static method: "Show me the books borrowed by John" (needs a specific member - John)
  • Static method: "How many total books does the library have?" (asks about the library itself, not any specific member)

You can ask about total books without knowing any specific library member. That's a static method!

üìÑ Student.h - Static Methods Example
class Student {
private:
    string name;
    int rollNumber;
    double gpa;
    
    static int totalStudents;
    static double totalGPA;
    static string universityName;

public:
    Student(string n, double g) {
        name = n;
        gpa = g;
        rollNumber = ++totalStudents;
        totalGPA += g;
    }

    // Regular method - needs an object
    void displayInfo() {
        cout << "Name: " << name << ", GPA: " << gpa << endl;
    }

    // Static method - works without an object!
    static double getAverageGPA() {
        if (totalStudents == 0) return 0.0;
        return totalGPA / totalStudents;
    }

    static void displayUniversityInfo() {
        cout << "University: " << universityName << endl;
        cout << "Total Students: " << totalStudents << endl;
        cout << "Average GPA: " << getAverageGPA() << endl;
    }

    static void setUniversityName(string name) {
        universityName = name;
    }
};
Static methods are called using the class name: Student::getAverageGPA() not student1.getAverageGPA(). Though the latter works, it's confusing because it suggests the method belongs to that specific object.

üìê Part 2: Class Diagrams - The Blueprint Language

What is a Class Diagram?

Imagine you're building a house. Before you start construction, you need blueprints, right? Class diagrams are the blueprints of software! They're a visual way to show the structure of your classes before you write any code.

Real-World Analogy: Restaurant Menu Design üìã

Before a restaurant opens, the chef doesn't just start cooking randomly. They create a menu (blueprint) that shows:

  • Dishes (Classes): Pizza, Burger, Pasta
  • Ingredients (Attributes): Size, toppings, price
  • Preparation methods (Methods): Bake, grill, serve
  • Relationships: "Combo Meal contains Burger + Fries + Drink"

Class diagrams do the same thing for your code!

// Based on the BankAccount UML diagram above

class BankAccount {
private:

        if (amount > 0 && amount <= balance) {
            balance -= amount;

    double getBalance() {
        return balance;
    }

    // Matches diagram: + displayAccount(): void

        cout << "Account #" << accountNumber
             << " | Holder: " << accountHolder
             << " | Balance: $" << balance << endl;
    }

    static int getTotalAccounts() {
        return totalAccounts;
    }

üé™ Part 3: Dynamic Objects - Memory on Demand

What is Dynamic Memory Allocation?

So far, we've been creating objects like this: BankAccount acc1;. But what if you don't know how many objects you need until the program runs? What if you need to create and destroy objects on-the-fly? That's where dynamic memory comes in!

Real-World Analogy: Restaurant Tables üçΩÔ∏è

Imagine two types of restaurants:

  • Fixed restaurant (Static Memory): Has exactly 20 tables. If 25 customers come, 5 are turned away. If only 10 come, 10 tables sit empty wasting space.
  • Dynamic restaurant (Dynamic Memory): Brings out tables ONLY when customers arrive and removes them when they leave. Efficient use of space!

Dynamic memory lets your program be the "smart restaurant" - creating objects only when needed!

Stack vs Heap Memory

Your program has two main memory areas:

Feature Stack Memory Heap Memory
Allocation Automatic (compiler does it) Manual (you do it with new)
Deallocation Automatic (when scope ends) Manual (you must use delete)
Size Small (usually ~1MB) Large (limited by system RAM)
Lifetime Ends with function/block Exists until you delete it
Speed Faster ‚ö° Slightly slower
Example Student s1; Student* s2 = new Student();

Creating Objects with new

The new operator allocates memory on the heap and returns a pointer to it.

üìÑ Static vs Dynamic Object Creation
#include <iostream>
using namespace std;

class Product {
private:
    string name;
    double price;

public:
    Product(string n, double p) : name(n), price(p) {
        cout << "‚úÖ Product created: " << name << endl;
    }

    ~Product() {
        cout << "‚ùå Product destroyed: " << name << endl;
    }

    void display() {
        cout << "Product: " << name << ", Price: $" << price << endl;
    }
};

int main() {
    // STATIC ALLOCATION (on Stack)
    cout << "=== Static Allocation ===" << endl;
    Product laptop("Laptop", 999.99);
    laptop.display();
    // laptop will be automatically destroyed at end of main()

    // DYNAMIC ALLOCATION (on Heap)
    cout << "\n=== Dynamic Allocation ===" << endl;
    Product* phone = new Product("Phone", 699.99);  // Creates on heap
    phone->display();  // Use arrow operator (->) with pointers
    
    // Must manually delete!
    delete phone;  // Frees memory and calls destructor
    phone = nullptr;  // Good practice: avoid dangling pointers

    cout << "\n=== End of main() ===" << endl;
    return 0;
}

/* OUTPUT:
=== Static Allocation ===
‚úÖ Product created: Laptop
Product: Laptop, Price: $999.99

=== Dynamic Allocation ===
‚úÖ Product created: Phone
Product: Phone, Price: $699.99
‚ùå Product destroyed: Phone

=== End of main() ===
‚ùå Product destroyed: Laptop
*/
Notice the order of destruction! The dynamic object (phone) is destroyed when we call delete, but the static object (laptop) is destroyed when main() ends.
If you forget to delete a dynamically allocated object, you create a memory leak! The memory stays occupied even though you can't use it anymore. It's like leaving restaurant tables out forever - eventually you run out of space!

Why Use Dynamic Memory? ü§î

üìÑ Example: Dynamic Array of Objects
#include <iostream>
using namespace std;

class Student {
private:
    string name;
    double gpa;

public:
    Student() : name("Unknown"), gpa(0.0) {}
    Student(string n, double g) : name(n), gpa(g) {}

    void display() {
        cout << name << ": " << gpa << endl;
    }
};

int main() {
    int numStudents;
    cout << "How many students? ";
    cin >> numStudents;

    // Create dynamic array based on user input!
    Student* students = new Student[numStudents];

    // Input student data
    for (int i = 0; i < numStudents; i++) {
        string name;
        double gpa;
        cout << "Enter name and GPA for student " << (i+1) << ": ";
        cin >> name >> gpa;
        students[i] = Student(name, gpa);
    }

    // Display all students
    cout << "\n=== All Students ===" << endl;
    for (int i = 0; i < numStudents; i++) {
        students[i].display();
    }

    // CRITICAL: Delete the array!
    delete[] students;  // Note the [] for arrays

    return 0;
}
When deleting arrays, use delete[] not just delete! This ensures all objects in the array are properly destroyed.

Common Mistakes and How to Avoid Them

‚ö†Ô∏è Common Pitfalls
// ‚ùå MISTAKE 1: Memory Leak
Product* p1 = new Product("Laptop", 999);
// ... use p1 ...
// Forgot to delete! Memory leak! ‚ùå

// ‚úÖ CORRECT
Product* p1 = new Product("Laptop", 999);
// ... use p1 ...
delete p1;
p1 = nullptr;

// ‚ùå MISTAKE 2: Dangling Pointer
Product* p2 = new Product("Phone", 699);
delete p2;
p2->display();  // ‚ùå Accessing deleted memory! Crash!

// ‚úÖ CORRECT
Product* p2 = new Product("Phone", 699);
delete p2;
p2 = nullptr;  // Now p2 points to nothing (safe)
if (p2 != nullptr) {  // Check before using
    p2->display();
}

// ‚ùå MISTAKE 3: Double Delete
Product* p3 = new Product("Tablet", 499);
delete p3;
delete p3;  // ‚ùå Deleting same memory twice! Crash!

// ‚úÖ CORRECT
Product* p3 = new Product("Tablet", 499);
delete p3;
p3 = nullptr;  // Prevents double delete

// ‚ùå MISTAKE 4: Wrong delete for arrays
Student* arr = new Student[5];
delete arr;  // ‚ùå Should use delete[]

// ‚úÖ CORRECT
Student* arr = new Student[5];
delete[] arr;  // ‚úÖ Proper array deletion

Best Practices for Dynamic Memory üéØ

  • Always pair new with delete: Every allocation needs a deallocation
  • Set pointers to nullptr after deleting: Prevents dangling pointers
  • Check for nullptr before using pointers: Prevents crashes
  • Use delete[] for arrays: Don't forget the brackets!
  • Consider using smart pointers (C++11+): Automatic memory management (you'll learn this later!)

üèóÔ∏è Part 4: Composition - Building Complex Objects

What is Composition?

Composition is one of the most powerful concepts in OOP! It's all about building complex objects by combining simpler objects. Think of it as building with LEGO blocks - each block is simple, but together they create something amazing!

Real-World Analogy: Your Computer üíª

Your computer is a perfect example of composition:

  • A Computer (whole) is composed of:
    • CPU (part)
    • RAM (part)
    • Hard Drive (part)
    • Graphics Card (part)
  • If you throw away the computer, all these parts go with it (strong "has-a" relationship)
  • The parts don't make sense without the whole - a CPU sitting alone isn't very useful!

In programming, we model this by having a Computer class that CONTAINS objects of CPU, RAM, etc.

Composition vs Inheritance

You'll learn about inheritance later, but here's a quick comparison:

Feature Composition (Has-A) Inheritance (Is-A)
Relationship One class CONTAINS another One class IS A TYPE of another
Example Car HAS-A Engine Dog IS-AN Animal
Lifetime Parts live and die with whole Independent lifetimes
Flexibility More flexible (change at runtime) Less flexible (fixed at compile time)
When to use Building complex from simple Specializing general concepts
Rule of thumb: Use composition when you can say "has-a" and inheritance when you can say "is-a". A car HAS-A engine (composition). A sports car IS-A car (inheritance).

class Wheel {

class Bicycle {

        frontWheel = new Wheel(wheelSize);
        rearWheel = new Wheel(wheelSize);
        cout << "üö≤ Bicycle assembled: " << brand << endl;
    }

    ~Bicycle() {
        cout << "üö≤ Destroying bicycle: " << brand << endl;
        // IMPORTANT: Delete dynamically created components!
        delete frontWheel;
        delete rearWheel;
    }

    void display() {
        cout << "Bicycle: " << brand << endl;
    }
};

int main() {
    cout << "=== Creating Bicycle ===" << endl;
    Bicycle bike("Mountain Pro", 26);
    bike.display();
    
    cout << "\n=== End of main (auto-destruction) ===" << endl;
    return 0;
}

When using dynamic composition, ALWAYS delete the components in the destructor! Otherwise you'll create memory leaks. The container object is responsible for cleaning up its parts.

Benefits of Composition üéØ

  • Reusability: Use the same component in different classes (Engine in Car, Boat, Plane)
  • Modularity: Change one component without affecting others
  • Encapsulation: Hide complex implementation details
  • Flexibility: Easy to swap out components
  • Maintainability: Fix bugs in one place, affects all users of that component
